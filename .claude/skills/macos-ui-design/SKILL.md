# macOS UI デザイン指針(SwiftUI)

出典: Apple HIG(macOS)と WWDC 2018「Designing Fluid Interfaces」。Web 向け翻訳である emilkowalski/skills の apple-design を検証し、SwiftUI/macOS に適用可能な原則のみをここに蒸留した(CSS / Pointer Events / Framer Motion 前提の実装ガイダンスは採用しない)。

## 原則

1. **標準コンポーネント優先。** `NavigationSplitView`、`.inspector`、`Menu`、`Table`、`.searchable` 等の標準セマンティクスを自作より優先する。標準が持つ挙動(状態保存、折りたたみ、アクセシビリティ、キーボード操作)を無償で得る。自作するのは標準で表現できないと確認できた場合のみで、その理由を PR に書く。
2. **応答性。** 入力には即座に視覚フィードバックを返す。時間のかかる処理(モデルダウンロード、要約生成、再文字起こし)は開始を即時に見せ、進捗を表示する。ボタンが「効いたかどうか分からない」状態を作らない。
3. **割り込み可能性。** アニメーションと遷移は途中で操作されても現在値から滑らかに再開できること。SwiftUI の spring アニメーションは既定で割り込み可能 ―― これを壊す固定 duration のキーフレーム連鎖や、完了を待ってから次を始める逐次遷移を避ける。
4. **直接操作。** ドラッグ・スクラブ(波形シーク等)中は、対象がポインタに吸い付いて追従すること。離した瞬間の速度をアニメーションに引き継ぐ(`DragGesture` の `velocity` を spring の初速へ)。

## アニメーション

- 基本は `.spring(duration: 0.3...0.4, bounce: 0)`。バウンスは既定で使わない。
- ユーザーの運動量を引き継ぐ場面(ドラッグ解放、シート)のみ `bounce: 0.2` まで許容。
- `withAnimation` の明示指定を優先し、暗黙アニメーションの重ね掛けで挙動が読めなくなるのを避ける。
- **Reduce Motion 対応**: `@Environment(\.accessibilityReduceMotion)` を確認し、有効時はスライド/スプリングを短いクロスフェード(opacity)に置き換える。

## レイアウトとマテリアル

- 階層は Material で表現する: `.regularMaterial` / `.thinMaterial` 等の標準マテリアルを使い、独自の半透明レイヤを組まない。
- ペイン構成は D3 / ADR-0008 の決定(NavigationSplitView 系)に従う。ペイン幅・折りたたみ状態は標準の復元に任せる。
- ウィンドウは macOS の慣習に従う: ツールバーは `.toolbar`、右サブペインは `.inspector`。

## タイポグラフィ

- システムフォント(SF Pro / ヒラギノ連携)+ セマンティックスタイル(`.largeTitle` 〜 `.caption`)のみ。固定ポイント指定は理由がある場合だけ。
- Dynamic Type を尊重する(固定フレームで文字を切らない)。
- 文字起こし本文は読み取りが主目的: 行間は既定より詰めない、volatile(未確定)行は `.secondary` で区別する(domain-pitfalls #6 の描画方針)。

## macOS の慣習

- 主要操作には ⌘ ショートカットを付ける(既存: 開始/停止 ⌘R、議事ログ ⌘L)。
- 右クリック(コンテキストメニュー)と選択ポップアップの両方から同じ操作に到達できるようにする。
- 破壊的操作(削除、文字起こしの上書き)は確認を挟み、可能なら Undo を提供する。
- メニューバーアイコンは状態(待機 / 録音中 / エラー)をテンプレート画像の変化で伝える。色だけに依存しない(アクセシビリティ)。

## チェックリスト(UI の PR 前)

- [ ] 標準コンポーネントで表現できないか検討したか(自作した場合、理由を PR に書いたか)
- [ ] 時間のかかる処理に即時フィードバックと進捗があるか
- [ ] アニメーションは spring(bounce 0)で、割り込みに耐えるか
- [ ] Reduce Motion で破綻しないか
- [ ] キーボード操作(ショートカット・フォーカス)で主要操作に到達できるか
- [ ] ライト/ダーク両方で確認したか
