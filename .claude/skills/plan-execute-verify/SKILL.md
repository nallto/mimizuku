# Plan → Execute → Verify ループ

## いつ使うか

- 複数ファイルに跨る変更、設計判断を含む変更、破壊的操作を含む作業のすべて。タイポ修正など自明な 1 行変更には不要。迷ったら使う。
- Mimizuku では、Core Audio taps や Speech framework に触れる変更は必ずこのループを使い、先に `docs/domain-pitfalls.md` を再読する。

## 手順

### 1. Plan(計画)

着手前に以下を簡潔に提示し、**承認を待つ**。承認前に編集・コミット・push をしない。

- 目的(何を・なぜ)
- 変更対象ファイルと変更内容の要約
- 検証方法(どのテスト・どのコマンドで正しさを確認するか)。ハードウェア/TCC 経路に触れる場合は、人間がローカルで検証すべき部分を明示する。
- トレードオフ・代替案(あれば pros/cons)
- 設計判断を含む場合: ADR 起票の要否

選択肢を提示する場合は、番号付きの選択式で問う。

### 2. Execute(実行)

- 計画の単位で実行する。逸脱が必要になったら、逸脱内容と理由を報告して再承認を得る。
- 実装・テスト・ドキュメントを同じ作業単位で更新する(AGENTS.md の不変条件)。

### 3. Verify(検証)— 2 段構え

検証は「機械検証」と「第三者検証」の 2 段で行う。**自分の仕事を自分だけで採点しない。**

1. **機械検証**: `just check` を実行し、green を確認する。
2. **第三者検証**: 非自明な変更では、verifier サブエージェント(`/verify`)に要求と diff を渡し、新鮮なコンテキストで敵対的にレビューさせる。verifier の FAIL は不合格として扱い、指摘を解消してから再検証する。

注意: `just check` はマイク / システム音声 / TCC 経路を実行できない。それらは計画で「人間がローカルで行う手動テスト」(soak run、デバイス切替、権限付与)を明示し、エージェントは計測用の仕掛け(ログ・診断画面)を用意する。

red / FAIL の場合: 原因を特定して修正し、再実行する。3 回修正しても解消しない場合は、状況を整理して報告し、指示を仰ぐ(無限に試行しない)。

### 4. Report(報告)

- 実施内容・検証結果(実行したコマンドと結果、verifier の判定、保留中の手動テスト)・残課題を報告する。
- 検証を通さずに「完了」と報告しない。

## 長時間タスクの状態ファイル

複数セッション・複数コンテキストに跨る作業では、計画と進捗をコンテキストではなく**ディスク上の作業ファイル**に持たせる(コンテキストは失われる前提)。

- 作業ブランチ内に `PLAN.md` 等を置き、「残ステップ・完了済み・判明した事実」を更新しながら進める。
- 作業ファイルは一時物であり、マージ前に削除する(squash 本文・docs・ADR に昇華させる)。恒久的な知識の置き場所は AGENTS.md の「知識の置き場所」に従う。
