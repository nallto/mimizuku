# ADR-0005: プロダクトスコープを拡張し、実装計画を「記録先行」の 6 フェーズ構成へ再編する

- ステータス: Accepted
- 日付: 2026-07-20
- 関連: #26 / ADR-0001, ADR-0003

## Context(背景)

旧計画([IMPLEMENTATION_PLAN.md](../plan/IMPLEMENTATION_PLAN.md) の旧版、Slice 0〜4)は「字幕ではなくライブ議事ログ」を中心に据え、録音(生音声の保存)は「任意・既定オフ」の付随機能だった。プロダクト要件を再定義した結果、次の機能群が求められることになった:

- 録音した音声の保存・再生(波形表示・再生位置ハイライト・速度変更)
- 3 ペインのメインウィンドウ(ファイル一覧 / 波形+文字起こし / 要約等サブペイン)
- タグ・ステータスによるファイル管理とフィルタ、横断検索
- アプリ内辞書による認識精度向上、録音ファイルからの再文字起こし
- リアルタイム翻訳(原文と訳文のペア表示)、要約・TODO 抽出
- 動画・音声ファイルのインポート
- 翻訳・要約はローカル/クラウド LLM へ差し替え可能な抽象を持つ

これは「ライブ議事ログ単機能」から「**録音・オンデバイス文字起こし・ノートを統合したアプリ**」への転換であり、録音は付随機能ではなく一級機能になる。旧 Slice 2〜4 の構成のままでは受け止められないため、計画の全面再編が必要。

開発順序について検討した代替案:

- **メイン画面 UI 先行**: 見た目と操作感を早期に固められるが、表示すべき実データ(保存されたセッション)が存在せず、UI 検証がモック頼みになる。却下。
- **リアルタイム体験先行**(文字起こし+翻訳の磨き込み): 録音・永続化を後回しにすると「使ったのに記録が残らない」期間が長引き、後段機能(再生・検索・要約)の土台も揃わない。却下。
- **記録の完成を先行**(採用): 録音 + システム音声 + 両ソース同時 + 永続化を最初に完成させる。「まず確実に記録が残る」状態を最短で作り、実データが貯まることで後段 UI・検索・AI 機能の検証を実データで行える。

## Decision(決定)

1. プロダクト定義を「録音・オンデバイス文字起こし・ノートを統合したメニューバー常駐アプリ」へ拡張する(README / AGENTS.md の概要を同一 PR で更新)。
2. **録音を一級機能とする。** 旧計画の「録音は任意・既定オフ」を廃し、セッションは音声ファイル + 文字起こしのセットとして保存される。
3. 実装計画を **6 フェーズ・20 スライス**へ再編する。順序は「記録の完成を先行」:
   - フェーズ 1: 記録の完成(録音・システム音声・デュアルストリーム・永続化)
   - フェーズ 2: メイン画面(3 ペイン・再生・波形・タグ/フィルタ・選択操作)
   - フェーズ 3: 検索・辞書
   - フェーズ 4: AI(翻訳・要約。Apple 標準フレームワークで開始し、プロトコル抽象で差し替え可能に)
   - フェーズ 5: ファイル入力(再文字起こし・インポート)
   - フェーズ 6: OSS リリース
4. 複数の選択肢がある論点は、対象スライスの前に**判断タスク**(選択肢比較 → ユーザー判断 → ADR 化)として独立に計画へ組み込む(D1〜D8)。
5. 翻訳は Translation framework、要約は FoundationModels(共に macOS 26 オンデバイス)で開始する。クラウド LLM への差し替え条件は判断タスク D7(ADR-0010 予定)で別途決定する — ハード制約「音声・文字起こしをデバイス外に出さない」との整合はそこで扱い、本 ADR では変更しない。
6. 旧 Slice 1(マイク→ライブ文字起こし)は新構成の Slice 1 と内容が一致するため、実装済みブランチをそのまま着地させる。旧 Slice 2〜4 の Issue(#10 #11 #12)はクローズし、新スライス構成で起票し直す。

## Consequences(結果)

- (+) 「まず確実に記録が残る」状態が最短で成立し、以降のスライスは常に実データで検証できる。
- (+) 要件→スライス対応表を計画に持つことで、要件の取りこぼしを機械的に検知できる。
- (+) 判断タスクの明示により、大きな分岐(保存形式・検索方式・AI 抽象)を実装前にユーザー判断へ載せるプロセスが固定される。
- (−) 録音一級化により、ディスク容量・ファイル形式・クラッシュ耐性(判断 D1)、プライバシー文書の更新など、旧計画にはなかった負担を引き受ける。
- (−) スコープ拡張により OSS リリース(フェーズ 6)は旧計画より後ろへずれる。リリース時期の再判断は別途行う。
- 本 ADR は方針転換の記録であり、個別の技術選定(保存形式・検索方式・AI 抽象等)は ADR-0006 以降に委ねる。フェーズ順序が成立しなくなる条件(例: SpeechAnalyzer の品質が実用未満と判明)では、フェーズ 1 のチェックポイント(ja-JP 品質評価)で計画を見直す。
