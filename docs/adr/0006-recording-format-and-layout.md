# ADR-0006: 録音は CAF(PCM)で書き、停止時に AAC へ変換する。ストリーム毎 2 ファイル

- ステータス: Accepted
- 日付: 2026-07-20
- 関連: #28(判断 D1)/ ADR-0005 / S2(#35)・S7(#40)・S18(#51)

## Context(背景)

録音は一級機能であり(ADR-0005)、S2 の受け入れ条件に「録音中の `kill -9` でもファイルが復元可能」がある。一方で日常的な会議利用(1〜3 時間/日)ではディスク消費が無視できない。前提: マイクは実質モノラル 48kHz、システム音声はステレオ 48kHz。

検討した代替案:

- **(a) CAF(PCM)のみ**: CAF は音声チャンク長を不定のまま追記できる設計で、突然死してもヘッダ修復なしに読める(WAV の RIFF ヘッダ更新問題が無い)。最小実装だが、PCM 16bit で両ストリーム合計約 1GB/h とディスク負担が大きい。却下(ただし段階実装の第一段として採用)。
- **(b) AAC(.m4a)直接書き込み**: 約 70〜90MB/h と最小サイズだが、MP4 はインデックス(moov)をファイル末尾に書くため異常終了で再生不能になり、復元対策の自作が必要。「記録の完成」の目的に反するため却下。
- **(c) 録音中 CAF + 停止時 AAC 変換**: 録音中の安全性は (a) と同一。ハードウェア AAC エンコーダにより変換は実時間よりはるかに速い。変換の状態管理が増えるのが代償。**採用**。

レイアウトは「ストリーム毎 2 ファイルのみ」と「+ 停止時ミックス生成」を比較し、ミックスは現時点で使う機能(共有・単純再生)が無く容量 1.5 倍になるため生成しない。再生の 2 ファイル同期は S7 で対応する(計画済み)。

## Decision(決定)

1. **録音中は CAF(LPCM 16bit・native サンプルレート、既定 48kHz)**で追記書き込みする。マイク=モノラル、システム音声=ステレオ。
2. **停止時に AAC(.m4a)へ変換**し、変換結果を検証(デコード可否とフレーム長の照合)してから CAF を削除する。ビットレート起点: マイク 64kbps モノラル / システム 128kbps ステレオ(実聴で調整可)。
3. **ストリーム毎 2 ファイルのみ**。ミックスファイルは生成しない(エクスポート/共有機能が必要になった時にその場で生成する)。
4. セッションディレクトリとファイル名:
   ```
   ~/Library/Application Support/Mimizuku/sessions/<yyyyMMdd-HHmmss>/
     mic.caf    → 変換後 mic.m4a
     system.caf → 変換後 system.m4a
     (文字起こし・メタデータの形式は D2 / ADR-0007 で決定)
   ```
5. **時刻原点**: セッション開始時刻 = 各録音ファイルの 0 秒 = `TranscriptSegment.start/end` の原点。各ストリームの実開始オフセット(捕捉開始の僅かな差)はメタデータに記録する(S7 の再生ハイライト同期の前提)。
6. クラッシュ回復: 起動時に `sessions/` 配下へ残存する `.caf` を検出したら、「未変換の録音」として再変換を提案する(録音中クラッシュの CAF はそのまま再生・文字起こし可能)。
7. S2 内の段階実装: まず (a)(CAF のみ)で経路を通し、変換(2.)と回復(6.)を同スライスの最終タスクとして追加する。変換が未実装でも (a) の状態でスライスは成立する。

## Consequences(結果)

- (+) 録音中のクラッシュ耐性が最大(S2 の `kill -9` 受け入れ条件を形式選定で担保)。
- (+) 保存後のディスク消費は AAC 水準(両ストリーム合計 約 70〜90MB/h)。
- (+) ストリーム分離により話者ラベル(S4)と個別再文字起こし(S18)がファイル構造でそのまま成立。
- (−) 変換パイプライン(変換済み検証 → CAF 削除)の状態管理が増える。変換中の一時的なディスク二重使用(PCM + AAC)を許容する。
- (−) 再生(S7)は 2 ファイル同期(`AVAudioPlayerNode` 2 本)が前提になる。
- (−) 再文字起こし(S18)は AAC デコード経由になる。音声認識への影響は実用上無視できる想定だが、S18 で品質差が観測されたら本 ADR を見直す。
- CAF のクラッシュ耐性の主張は S2 の受け入れテスト(`kill -9` → 復元確認)で実証する。実証できなければ本 ADR を再検討する。
