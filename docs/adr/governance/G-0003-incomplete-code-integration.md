# G-0003: 未完成コードの main 統合は keystone を優先し、flag は削除期限つきで使う

- ステータス: Accepted
- 日付: 2026-07-13
- 関連: G-0001

## Context(背景)

trunk-based development(G-0001)では、大きな変更も小さな PR の列として main に統合し続ける。このとき「未完成のコード(dark code)が本番に入る」ことへの対処が必要になる。管理を失ったdark code は dead code に転化し、解析の阻害・事故(死んだコードパスの誤作動)の原因になる。一方、長寿命 feature ブランチは巨大コンフリクト・レビュー不能な差分・遅いフィードバックという、より大きい統合リスクを持つ。

## Decision(決定)

未完成状態が main に存在する期間が生じる変更は、次の優先順位で統合方法を選ぶ:

1. **PR 分割**: 1〜2 日で main に入る単位に分解できるなら、特別な技法は使わない(第一選択)。
2. **Keystone pattern**: 新機能は入口(UI・エンドポイント・エントリポイント)を最後の PR で接続する。入口がなければ到達不能であり、flag は不要。
3. **Branch by abstraction / expand–contract**: 既存機構の置き換え・スキーマ移行は、抽象層の導入または「拡張 → 両対応 → 収縮」の段階移行で行う。
4. **Feature flag**: 段階的ロールアウト・A/B・即時 kill switch が**要件として**必要な場合のみ使う。「未完成を隠す」目的の第一選択にしない。

Feature flag を導入する場合のフラグ衛生(必須):

- フラグにはオーナーと削除期限(または削除条件)を必ず設定する。
- 導入 PR と同時に削除用 Issue を起票する(PR テンプレートのチェック項目)。
- リリース用フラグ(短命)と運用スイッチ(長命)を区別して命名・管理する。
- リリース用フラグは全量リリース後 4 週間以内に削除 PR を出す。
- 同時に active なリリース用フラグは 5 個まで(組み合わせ爆発の抑制)。

## Consequences(結果)

- 得るもの: 長寿命ブランチの統合リスクを回避しつつ、dark code の dead code 化を工程で防ぐ /「レビュー済み・CI 済みの PR の列」として履歴が残る。
- 失うもの: keystone / 抽象層の設計に一手間かかる。flag を使う場合は分岐が一時的に増える。
- 備考: stale フラグの検出と削除 PR の作成は定型作業であり、AI エージェントへの定期タスクとして委任しやすい。
