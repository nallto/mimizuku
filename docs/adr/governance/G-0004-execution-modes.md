# G-0004: 実行モードは承認駆動を既定とし、ループ実行は条件とガードレールを満たす場合に使う

- ステータス: Accepted
- 日付: 2026-07-13
- 関連: G-0001, G-0003, ADR-0001

## Context(背景)

Claude Code には、人間が各ターンを承認しながら進める使い方と、停止条件を満たすまで自律的に反復するループ実行(`/goal`・`/loop`・`/schedule`・auto mode)がある。モード選択を都度の空気読みに任せると判断が揺れる(G-0001 と同じ問題構造)。上限のない無人ループは、トークン消費の暴走・検証されない変更の蓄積という具体的なダウンサイドを持つ。

**Mimizuku 固有の制約(ここでは決定的):** 核となる作業 ―― Core Audio taps、Speech framework、TCC 権限フロー、実音声ハードウェア ―― は CI でもサンドボックスでも実行できない(domain-pitfalls #8)。したがって「ネットワーク制限サンドボックス内で自律実行」というモデルは本プロジェクトの核に適用できない。捕捉・文字起こしスライスの受け入れは本質的に人間による実機テストを要する。

## Decision(決定)

**既定は承認駆動**(計画 → 承認 → 実行 → 2 段検証(`just check` + verifier)→ 報告)。探索・設計判断・要求が曖昧な作業、そして**すべての音声・TCC・ハードウェア依存の作業**はこのモード。

ループ実行は、以下の**条件とガードレールをすべて満たす場合に**、かつ CI が実際に検証できる純ロジック部分(モデル・エンコード・ルーティング・watchdog ロジック)に限って使う:

| 類型 | 使う条件 | 必須ガードレール |
|---|---|---|
| goal-based(`/goal`) | 完了条件が機械検証可能(例: 「テスト N 本 green」) | ターン上限を必ず指定。曖昧な goal を渡さない |
| time-based(`/loop`/`/schedule`) | 外部システムの監視・定期作業(CI 監視・トリアージ) | 間隔は監視対象の変化頻度に合わせる |
| proactive(auto mode) | 定型化された反復の流れが確立している | 純ロジック範囲のみ。捕捉/Speech/TCC 経路には使わない。小範囲で先にパイロット |

モードに関わらず不変のルール:

1. **検証の 2 段構えは維持する。** ループ内でも done の判定は `just check` + verifier (新鮮なコンテキストの第三者レビュー)を通す。
2. **G-0001 の統合フローは維持する。** ループでも main への直接コミットはせず、ブランチ → PR → CI green → squash。無人ループに main を触らせない。
3. **トークン消費を管理する。** 実行前に完了条件と上限を明示し、大規模実行の前に小範囲でパイロットし、`/usage` で消費を確認する。
4. 長時間・複数コンテキストの作業は状態をディスク(作業ファイル)に持たせる。

## Consequences(結果)

- 得るもの: モード選択が条件表で決まり判断が揺れない / 無人実行の事故経路(無上限ループ・未検証コミット・サンドボックス外実行)を事前に遮断できる。
- 失うもの: ループで済む単純作業にも初回はガードレール整備の手間がかかる。
- 見直し条件: 公式のループ関連機能の仕様変更時、または Apple が macOS ネイティブ経路を CI で実行する手段を提供した時に本 ADR を見直す。
